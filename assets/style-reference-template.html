🧭从第一性原理看 Agent Loop

LLM 本质上只是“根据上下文预测下一段输出”。要让它成为 Agent，必须补上 行动、反馈、停止 这三个环节。

ReAct解决“不知道去哪找”
Plan-and-Solve解决“步骤容易乱”
Reflection解决“第一版不够好”
CORE IDEA

🔁一次模型调用，为什么不等于 Agent？

第一性原理很简单：单次调用没有持续目标、没有外部反馈、没有错误恢复。Agent 的核心，是把模型放进一个可控的 反馈回路。

🧠LLM只能基于当前上下文生成输出
+
🔄Loop把输出变成行动，再把反馈放回上下文
=
⚙️Agent 行为会探索、会拆解、会修正
状态如何流动
工具如何接入
循环何时停止
WHY BUILD BY HAND

框架隐藏的，正是 Agent 会坏的地方

问题不是“会不会调 API”，而是失控时你知不知道哪里坏了。

高抽象框架会让 demo 很快跑起来，但也会把解析、重试、记忆和停止策略藏起来。

!
解析问题模型输出不稳定，工具调用就会失败。
↻
重试问题工具报错后，需要降级、重试或换路径。
⌁
上下文问题历史记录太乱，下一步决策就会偏。
■
停止问题没有边界，Agent 会越跑越远。
REACT

ReAct：用外部反馈对抗幻觉

当模型内部知识不够时，改进点不是“让它更自信”，而是让它 先想、再查、再根据结果修正。

💭Thought把“我缺什么信息”说清楚。
🛠️Action调用搜索、API、计算器或业务工具。
👁️Observation把工具结果放回上下文，修正下一步。
解决的问题

纯 CoT 会幻想，纯 Action 不会纠错。

改进点

清晰工具说明、稳定输出协议、高质量 Observation、max_steps。

PLAN-AND-SOLVE

Plan-and-Solve：先降低问题复杂度

当任务路径相对确定，最大风险不是缺信息，而是 跳步、漏步、顺序乱。所以先规划，再执行。

🧩
Plan把大问题拆成小步骤。
▶
Solve按计划逐步执行。
✓
Check发现计划错误时要能回退。
ReAct

世界不清楚，边观察边改。问题在外部环境里。

Plan-and-Solve

题目已清楚，先拆解再做。问题在内部结构里。

REFLECTION

Reflection：把质量改进显式化

当第一版答案通常不够好，改进点不是盲目重跑，而是加入 评价标准、反馈、修订。

✍️ DRAFT先产出

没有初稿，就没有可检查对象。

🧪 CRITIQUE再批改

用标准指出错误、遗漏和低效之处。

✨ REFINE再修订

根据反馈重写；必要时继续循环。

CHOOSING A LOOP

🎯三种范式怎么选？看问题的形状

不是哪个更高级

从第一性原理看，范式选择就是在问：当前任务最缺的是信息、结构，还是质量反馈？

缺外部信息ReAct

边看边决定。

缺拆解结构Plan-and-Solve

先想清楚再执行。

缺质量反馈Reflection

先做出来再优化。

LANGGRAPH BRIDGE

🕸️LangGraph 不是新概念，是把 loop 工程化

当 loop 变复杂，就需要把状态、节点、工具、边和停止条件显式管理起来。

State上下文如何保存
Node每一步谁决策
Tool如何连接环境
Edge下一步去哪
Stop循环何时结束
改进点：把隐式 prompt 链，变成可观察、可调试、可终止的图。
FIVE TAKEAWAYS

这章最该记住的 5 句话

1Agent = LLM + 状态 + 工具 + 反馈 + 停止条件。
2ReAct 改进信息获取，降低幻觉。
3Plan 改进任务拆解，减少跳步。
4Reflection 改进最终质量，但要有评价标准。
5学框架之前，先看懂 loop 的失控点。
CLOSING

从会用 Agent，到能设计循环

真正的迁移能力，不是背三个范式，而是看见任务的问题形状，再选择合适的 loop。

ReAct先去看，再决定
Plan-and-Solve先想清楚，再去做
Reflection先做出来，再优化
